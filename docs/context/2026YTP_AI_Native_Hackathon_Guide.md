# 🚀 2026 YTP 台北未來黑客松：原生 AI (AI-Native) 開發與創新指引

> **「用 AI 懂台北，讓城市更懂你。」**
> 
> 這是一份專為「2026 YTP 黑客松」參賽團隊準備的實戰策略建議。我們鼓勵參賽團隊跳脫「僅在系統掛載對話框」的思維，轉而探索如何將大語言模型 (LLM) 深度整合進應用程式的核心邏輯。這份指引旨在協助各位運用**「原生 AI (AI-Native)」** 的力量解決台北市的真實痛點，並創造足以驚艷使用者的智慧體驗。

---

## 🏗️ 核心實踐：團隊開發的 Best Practices

在 32 小時的短跑競賽中，成功的關鍵往往在於如何有效地利用 AI 的推理與語意理解能力，而非耗費大量精力手刻死板的 `if-else` 或 Regex 邏輯。以下是我們推薦參賽團隊優先探索的四大 **「AI-Native 典範實踐」**：

### 1. 拋棄死板選單，擁抱「意圖驅動 (Intent-Driven UI)」
傳統 Web 應用往往要求使用者點選多層選單才能得到結果；而 AI-Native 應用則能讓使用者透過一個簡單的文字框或語音指令完成所有互動。

*   **建議方向**：充分利用 LLM 的 **Structured Output (強制結構化 JSON 輸出)** 特性。讓 AI 聽懂使用者的口語化情境（例如：「大雨滂沱，想找安靜的地方躲兩小時雨讀點書」），並精準萃取出背後的實體（時間、天氣、地點）、情緒與動機參數。
*   **使用者價值**：這不僅僅是聊天，更是讓系統能透過自然語言即時驅動 UI/UX 的按鈕亮起或環境變換，為使用者帶來**「系統瞬間懂我」**的 Agentic 互動體驗。

### 2. 超越關鍵字搜尋，實作「語意檢索 (Semantic RAG)」
相較於傳統 SQL 的關鍵字匹配，我們更推薦運用向量技術來處理台北市的高維度資源。

*   **建議方向**：您可以考慮導入輕量級向量資料庫 (如 ChromaDB, Qdrant)，將台北市的數萬筆景點介紹、商圈動態，甚至政府 Open Data，預先透過 Embedding 模型化為數值向量。
*   **優勢揭秘**：這能讓系統呈現「理解力」。即使資料標籤中沒有特定關鍵字，系統也能透過相似度演算，為想體驗「台北歷史與潮流碰撞」的使用者推薦松山文創與四四南村。這是在 32 小時內打造深度推薦引擎的最強利器。

### 3. 建構系統的「不死鳥韌性 (Bulletproof Resilience)」
黑客松現場的網路環境往往富具挑戰，API 也可能遭遇頻率限制。確保系統能「優雅降級」展現了開發團隊的成熟度。

*   **建議方向**：在與 LLM 的通訊層 (Adapter) 中，實作完善的錯誤捕捉與分級機制。
*   **最佳實踐**：建議優先嘗試效能最強的模型；若遇 Rate Limit (HTTP 429)，可自動切換至輕量快速模型 (如 `gemini-1.5-flash`)；在完全斷網的極端情況下，確保系統有一套能讓 UI 正常渲染的 **Mock JSON 備援**。這將確保您的作品在任何情境下都能維持流暢。

### 4. 運用「無腳本智慧測試 (Subagent-Driven Validation)」
在分秒筆爭的競賽中，手寫繁雜的測試腳本是非常耗時的。

*   **建議方向**：推薦嘗試活用開發工具中的 **Browser Subagent** 或具備自覺能力的測試智能體。
*   **價值體現**：您可以用白話文告訴 AI 指令：「請在首頁輸入『東區宵夜』並驗證卡片是否正常產出」。讓 AI 成為您的專屬 QA 夥伴，將寶貴的開發時間投注在最重要的產品價值與核心邏輯上。

## 🛠️ 戰前整備：參賽者的實戰準備清單 (Pre-game Checklist)

要在 32 小時內產出高品質的 AI 原生應用，開發時間的每一秒都必須花在刀口上。我們建議團隊在比賽計時開始前，先行完成以下整備：

### 1. 策略性資源佈署 (API & Quotas)
最令人扼腕的情況是有了絕佳點子，卻卡在 API 額度限制。
*   **高配額備案**：除了主流模型，強烈建議熟悉具備高 Free Tier 額度的模型（例如 `gemini-1.5-flash` 提供驚人的每分鐘請求數），確保 Demo 過程中不會因 HTTP 429 錯誤而中途斷火；同時，建置本地端的向量資料庫（如 ChromaDB）能讓您的檢索功能更為穩定且不受限。

### 2. 工程效能自動化 (Hero Boilerplates)
黑客松不該把前 15 個小時花在「搞懂框架設定」。
*   **精煉您的 Starter Kit**：我們建議團隊提早選定並精通一套「AI 友善」的技術棧模版。不論是附帶 Pydantic 強制型別、已寫妥 LLM 適配器的 FastAPI 後端，或是具備語音辨識與精美 UI 組件的 Frontend，這些「英雄鷹架」能讓您在拿到題目的第一秒，就直接開始把腦中的 Prompt 轉化為程式碼。

### 3. 資料先行策略 (Curated Data Preparation)
AI 的精準度取決於資料。如果您在計時開始後才花一整天洗資料，這將失去競爭力。
*   **預處理與 Embedding**：建議團隊針對可能的台北市相關資料（如商圈資訊、即時天氣接口規格、甚至社群討論樣本）進行事前清洗與 Embedding 預處理。將其預鑄為 ChormDB 的向量檔，這能讓您在比賽中跨越「爬蟲與清洗」的坑洞，直接釋放驚人的檢索創意。

### 4. 高品質交付自我檢核 (Quality Dimensions)
與其猜測評審品味，不如用 AI 原生世代的標準自我要求：
*   **意圖精準度**：您的系統是否能捕捉到隱含的語意動機？
*   **架構韌性**：在模擬斷網或 API 失效的情境下，您的系統能「優雅降級」而不崩潰嗎？
*   **資料合成創新**：您是否利用 RAG 技術完美融匯了硬性的政府開放資料與感性的社群風向，產出連在地人都驚艷的洞察？

### 5. 活用 AI 導師 (Leverage AI Agents)
當 LLM 傳回來的 JSON 格式破裂，或模型陷入幻覺時，不要獨自崩潰。
*   **破障策略**：學習指派開發工具內建的專屬導師（如 Antigravity 這樣的 AI Agent）在比賽中期協助排錯。掌握 Prompt Engineering 的修補技巧與 Pydantic 結構化資料的驗證方法，將是您的團隊能快速突破瓶頸的祕技。
