# 開放資料源取得與防爬蟲對抗策略 (Data Acquisition Strategy)

**日期**：2026-03-23  
**主題**：臺北時光機 (Taipei Time Machine) - 從 `data.taipei` 轉移至 `TDX` 的實戰經驗與替代方案釋疑說帖

---

## 1. 核心困境：為何放棄臺北旅遊網 (data.taipei)？

在專案開發初期 (Phase 4)，我們曾依賴臺北市政府觀傳局提供的 `data.taipei` API 作為核心景點資料來源。然而在進入巨量資料匯入階段 (Phase 11-12) 時，我們遭遇了不可逆的技術阻礙：

*   **Cloudflare WAF (Web Application Firewall) 全面封鎖**：
    `data.taipei` 的對外端點啟動了極為嚴格的 Cloudflare 機器人防護機制 (Bot Fight Mode & Turnstile)。當系統使用標準的 Python HTTP 請求庫 (如 `requests` 或 `aiohttp`) 嘗試拉取 JSON 資料時，全數被伺服器以 `403 Forbidden` 或 `503 Service Temporarily Unavailable` 攔截。
*   **無法程式化存取**：這代表每一次請求都需要瀏覽器環境去解析並回傳複雜的 JavaScript 挑戰 (Challenge)，導致單純的後端 API 呼叫徹底失去作用。

為了確保黑客松系統展演的生命力與資料更新的自動化，我們**果斷決定放棄 `data.taipei`**，尋求更穩定的國家級基礎設施。

---

## 2. 破局之道：轉換至交通部 TDX 平台與 GPS 邊界框列策略

我們轉而全面擁抱 **交通部運輸資料流通服務 (TDX)**。TDX 提供明確的 API Key 機制 (OData)，是專為機關與企業應用程式設計的流通平台，沒有惱人的 Cloudflare 網頁驗證碼。

但 TDX 的「臺北市 (Taipei)」分類在某些端點（如：特色餐飲與在地活動）的資料量嚴重不足，甚至回傳 0 筆。我們採用了以下**架構轉換策略**來完美補足：

*   **全國拉取 + Bounding Box 過濾 (Geo-Spatial Filtering)**：
    我們不再受限於僵化的「城市代碼」。系統改為向 TDX 發出**全國性 (Nationwide)** 的最大限制筆數請求 (`$top=5000`)。
*   **實例作法**：在 Python 腳本中，我們對取回的全國資料進行經緯度 (Lat/Lng) 的範圍比對：
    ```python
    # 臺北生活圈 Bounding Box
    lat_min, lat_max = 24.9, 25.22  
    lng_min, lng_max = 121.4, 121.7 
    # 只保留落入此框內的景點
    ```
*   **成果驗收**：這個作法因禍得福，我們不僅找回了臺北市應有的龐大資料，甚至將新北市接壤的重要商圈也一併劃入！這使得系統最終能擁有多達 **1,792 筆** 高品質、具備地理脈絡的即時 POI 資料，徹底超越了最初規劃的規模。

## 3. 未來備案：如果必須拿下被保護的資料，有何替代方案？

雖然我們現在有一條順暢的 TDX 高速公路，但針對未來可能遇到「只有特定受 Cloudflare 保護的網站才有」的獨家資料（例如：社群封閉論壇、米其林在地隱藏版名單），本專案也明確梳理了以下四種技術突破備案，供後續擴充時直接調用：

### 備案 A. 轉向政府開放資料平台 (data.gov.tw) - (首選，合法免費)
地方單位的即時網站通常有防火牆，但他們必須定期將資料 Dump 上傳到國發會的 [政府資料開放平臺](https://data.gov.tw/)。
*   **作法**：直接對接 `data.gov.tw` 的靜態 CSV/JSON 連結下載，該平台完全沒有防爬蟲機制，是最穩當的退路。

### 備案 B. 第三方 Web Scraping API (開發極快，需付費)
業界如 ZenRows、ScraperAPI 等專精於代理突破的服務。
*   **作法**：將原本打向 `data.taipei` 的請求，轉發給 ZenRows 的 API。由他們的伺服器叢集代為執行瀏覽器挑戰，並回傳乾淨的 JSON。
*   **適用場景**：專案進入商轉，需要保證 100% SLA 的資料獲取。

### 備案 C. Headless Browser (無頭瀏覽器 + Stealth 擴充) - (免成本，維護重)
不依賴輕量化的 `requests`，改為啟動完整的虛擬瀏覽器。
*   **作法**：導入 Playwright 搭配 `playwright-stealth` 繞過反爬蟲機制，並透過監聽瀏覽器內的網路封包 (`page.on("response")`) 直接截取後方傳來的 JSON，而不必剖析厚重的 HTML DOM。
*   **特點**：開發成本較高，且爬蟲速度較慢，但非常適合應對每層防護都拉滿的終極關卡。

### 備案 D. TLS 指紋偽裝 (TLS Fingerprinting)
現代防火牆不僅檢查 User-Agent，還會檢查底層 SSL/TLS 握手的封包特徵 (JA3 Fingerprints)。
*   **作法**：在 Python 端實作改用 `curl-cffi` 套件，讓 HTTP 請求散發出「原生 Chrome 瀏覽器」的加密握手特徵，而非 Python 預設的 OpenSSL 特徵。
*   **適用場景**：作為不更改既有程式架構下，成本最低的第一次嘗試。

## 4. Phase 32 進化：從開放資料到菁英白名單 (Elite Whitelist)

隨著系統進入 Phase 32，我們發現單純依靠 TDX 開放資料，會導致推薦結果過度偏向「交通節點」與「大型觀光區」，而忽略了城市中最具靈魂的巷弄美食與最高端的體驗。

為此，我們的資料取得策略迎來了第二次重大轉變：
1. **OSM (OpenStreetMap) 深掘**：放棄無法取得深度資訊的政府 API，改為利用 OSM 的 Overpass API，以極高密度抓取單一行政區的所有小店，做為 LBS (Location-Based Service) 的降級備援。
2. **人工菁英注入 (Manual Injection)**：針對 AI 無法判斷的「米其林餐廳」、「頂級星級飯店」與「國家級地標」，我們寫入了 `scripts/elevate_*.py` 系列腳本，直接將超過 130 筆**世界級地標**強行寫入關聯資料庫，並標註最高權重的 `Tier 1 / Tier 2` 星級。

**最終結果**：透過自動化爬蟲與人工品味把關的「混血 (Hybrid) 擷取策略」，臺北時光機的資料庫成功凝鍊為 **2,367 筆 菁英地標**，達成大台北 119 站捷運站皆有高品質美食覆蓋的驚人成就。

---

**總結宣言**：
在資料即石油的年代，遇上 Cloudflare WAF 是常態。臺北時光機的架構轉換，從「卡在防護牆外」轉而利用「巨量地理資料裁切演算法」，再進化到「米其林菁英白名單」，建立了一道不對稱的資料護城河。這不但凸顯了我們團隊在遇到系統瓶頸時的**架構調度韌性**，更使得最終產品的涵蓋範圍與品味達到前所未有的深度。
