# 台北時光機：一趟資料的蛻變與旅程 (The Data Journey)

**日期**：2026-03-23  
**主題**：配合系統 DFD (Data Flow Diagram)，追蹤一筆使用者請求如何穿越 SQLite 關係檢索與 ChromaDB 向量檢索，最終化為推薦魔法的真實足跡。

---

## 序幕：使用者的呼喚 (The Input)

故事從一個平凡的下午開始。使用者人在**台北市政府（信義區一級戰區）**，看著窗外的大雨，打開了我們的 Web App，對著麥克風說出了一句充滿上下文考驗的需求：

> 👤 **User Query**: `「下著暴雨，我在台北市政府，我想找個不會淋濕的地方看展覽喝咖啡。」`

這句話包含了「天氣狀況 (暴雨)」、「地點需求 (室內)」、「活動意圖 (看展覽)」、「餐飲意圖 (喝咖啡)」。傳統的資料庫系統，如果沒有人為先去勾選「展覽」或「咖啡廳」的類別下拉選單，這句話在系統中只是一堆無意義的字元。但在臺北時光機的架構中，一段精彩的解析旅程才正要展開。

---

## 第一站：大腦的轉譯 —— 意圖解析 (Intent Parser)

使用者發出的請求字串，首先抵達了 **Intent Parser (Gemini Flash-Lite)**。
在這裡，專案的底層 `LLMAdapter` 設計發揮了作用。我們透過了 Pydantic 強制輸出的格式，要求模型拋棄人類對話的廢話，專注淬鍊出系統能理解的「語意標籤 (Semantic Tags)」。

*   ⚙️ **系統日誌**:
    `➤ 意圖模型解析出以下標籤: ["室內避雨", "藝文展覽", "咖啡輕食"]`

這是一次關鍵的蛻變！原本口語化的句子，被解構成了三個強烈的情境特徵。

---

## 第二站：升維打擊 —— 特徵向量化 (Embedding Encoding)

拿到標籤與使用者的原話後，我們不能直接去 SQLite 裡面寫 `SELECT * WHERE description LIKE '%室內避雨%'`。因為沒有任何一篇政府官方的景點介紹會寫自己是「室內避雨」的景點。

因此，資料流來到了 **Gemini-embedding-001** 模型。

*   ⚙️ **系統日誌**:
    `➤ 將字串打包輸入 Embedding 模型: '室內避雨 藝文展覽 咖啡輕食 下著暴雨，我想找個不會淋濕的地方看展覽喝咖啡'`
    `➤ 瞬間回傳了 768 維度的浮點數陣列 (Vector Representation)。`

在這 768 個維度的數學空間裡，這句話不再是文字，而是一個座標點。這個座標點位在「天氣惡劣」、「充滿藝術性」、「具備咖啡香氣」的空間交集處。

---

## 第三站：千軍萬馬中取敵首級 —— ChromaDB 向量配對

拿到了由使用者需求轉換而成的這把「數學座標鑰匙」，資料流衝進了我們的 **ChromaDB 向量資料庫**。

前兩天的大匯入 (Phase 11-12) 以及後來 Phase 33 的菁英白名單擴充，讓這裡沉澱了 **2,367 筆** 高品質...向量由模型更新後均為 768 維度。ChromaDB 在不到 3 毫秒的時間內，計算了輸入座標與 2,367 個地點座標的 **Cosine Similarity (餘弦相似度)**，並精準抓出了最近的點：

*   🎯 **真實 ChromaDB 模擬結果 (尚未套用地理 GPS)**:
    1.  🥇 `[語意距離 0.284]` **石碇淡蘭藝文館**
    2.  🥈 `[語意距離 0.290]` **板橋435藝文特區**
    3.  🥉 `[語意距離 0.298]` **世界宗教博物館 (永和)**

**【技術檢討：ChromaDB vs SQLite】**
你看到了嗎？**沒有任何傳統資料庫能做到這件事**。
在第一階段，ChromaDB 打破了資料來源的分類藩籬。它不管你是「餐廳(food)」還是「景點(spot)」，只要你散發出的語意特質同時具有「藝術 (特區/文物館)」與「可以避雨」，你就會被排在前面。這是全文搜尋 (ElasticSearch) 或者傳統關聯查詢 (SQLite) 絕對無法觸及的魔法領域。

---

## 第四站：照妖鏡與守門員 —— SQLite 實體過濾與距離矩陣

ChromaDB 非常聰明，但它缺乏**真實物理世界**的概念。「石碇淡蘭藝文館」在語意上確實非常符合「室內避雨、藝文」，但它遠在新北市石碇區！如果系統把它推薦給人在信義區「台北市政府」躲暴雨的你，這顯然是一個災難。

這時候，資料流進入了 **SQLite** 的管轄範圍，並與前端傳入的真實 GPS 座標展開激烈的計算：

*   ⚙️ **系統日誌**:
    `➤ 將 ChromaDB 初步挑選的大量景點 ID 交給 SQLite。`
    `➤ 利用使用者的 GPS (25.039, 121.564) 計算實體距離 (Haversine Distance)。`
    `➤ [扣分懲罰機制啟動]：世界宗教博物館與板橋 435 藝文特區，因為距離過遠，遭到強烈的降分淘汰。`
    `➤ [菁英提拔機制 Tier System]：位於市府大樓內的「臺北市政府 (台北探索館)」與距離僅 0.53km 的「中油藝廊」，因為距離極近且極具室內展覽性質，穩穩守住前排。`
    `➤ 從 SQLite 提取這些勝出者的官方 image_url 與完整的 address 資訊。`

**【技術檢討：為何我們需要雙料庫合擊】**
這就是這套架構最迷人的地方。SQLite 就像是現實世界的錨點。它用極低的運算成本加上距離懲罰演算法，把飄在雲端的高維度語意（ChromaDB）**強行拉回地面**。確保推薦名單不仅懂你，而且「走的到」。只有通過了物理距離檢驗的地點，才算真正活下來。而 Tier 菁英加權，則確保了使用者不會只看到身邊「最近卻很無聊」的地標。

---

## 終站：文字的魔術師 —— 導覽生成器 (Guide Generator)

最後，這些通過了 ChromaDB (語意吻合) 與 SQLite (距離吻合 + Tier 提拔) 雙重考核的最終勝者 —— **臺北市政府 (台北探索館)** 與 **中油藝廊**，連同它們的高畫質圖片網址，被送回給了 **Gemini Flash (Generator)**。

此時的 Generator 就像一位資深導遊，他結合了你所在的暴雨情境，為你寫下了溫暖的回覆：

> 🪄 **System Output**: 
> 「外面大雨滂沱別擔心！這份行程為您精選了信義區的室內藝文空間，讓您在舒適的環境中避雨，同時享受知性與藝術的洗禮，輕鬆度過悠閒的雨天時光，完全不用擔心淋濕！」

伴隨著這段文字，前端 UI 渲染出了精美的卡片，其中包含以下推薦理由：

*   **臺北市政府 (台北探索館)**：*「適合喜歡知性展覽的你。」*
*   **中油藝廊**：*「這是一個隱藏版的藝文小天地，展出風格多元且充滿藝術氣息。在雨天躲進這裡靜靜欣賞畫作或攝影，能讓心情瞬間平靜下來，享受難得的文青時光。」*

---

## 隱藏站：微交通情報網 —— YouBike 動態情境掛載 (Micro-Mobility Layer)

這時候行程卡片已經快要印出來了，但我們還差最後一哩路：「怎麼去？」
系統在把資料還給前端的最後一刻，呼叫了 **UBikeService**。

> 系統：*「等等！中油藝廊離使用者 500 公尺！他們在下大雨，但如果有人想拼一下腳踏車呢？或者等雨停後怎麼移動？」*

系統的記憶體快取 (TTL Cache) 中，隨時保有全臺北 1,749 個 YouBike 站點最新的資料。它透過 Haversine 演算法，瞬間抽出：
1. **起點**：「捷運市政府站(3號出口)」還有 12 輛車可借。
2. **終點**：「松仁路」站點還有空位可還。

這些資訊被封裝進 `poi.ubike_info` 中，化為前端的「綠色微交通小幫手」動態 UI，隨同卡片一起送到使用者的手中。

使用者的這趟資料旅程，正式畫下最完美且實用的句點。
