# 台北時光機：在地小店的逆襲與 RAG 雙引擎旅程 (The Hybrid Data Journey)

**日期**：2026-03-24  
**主題**：配合系統最新升級的 Hybrid LBS + RAG 雙引擎架構，追蹤一筆**被 ChromaDB 遺棄的在地小店**，如何透過 SQLite 的極限救援與 Threshold 評分機制，最終逆襲成為推薦榜首的真實足跡。

---

## 序幕：使用者的呼喚 (The Input)

故事從一次充滿生活感的日常查詢開始。使用者剛步出捷運站，想找個地方坐坐，於是對著我們的系統輸入了這樣一句話：

> 👤 **User Query**: `「捷運三民站附近的咖啡店」`

這句話看似簡單，但在我們擴充了 **15,230 筆** 全台北市的巨量資料庫之後，這卻是最容易讓純 AI 系統「翻車」的毀滅性測試。一趟驚險的資料拯救旅程，就此展開。

---

## 第一站：大腦的轉譯 —— 意圖解析 (Intent Parser)

使用者的請求首先抵達了 **Intent Parser**。
我們透過 LLM 將這句口語化的句子，解構成了系統的檢索關鍵字：

*   ⚙️ **系統日誌**:
    `➤ 意圖模型解析出目標類別: ["food", "Restaurant"]`
    `➤ 提取出字串關鍵字: "咖啡"`

到這個階段，系統準備好要去找「餐廳」類別中，與「咖啡」相關的資料了。

---

## 第二站：高維空間的階級霸凌 —— ChromaDB 的盲點

拿到標籤後，資料流進入了 **ChromaDB 向量資料庫**。

這裡存放了所有 15,230 筆景點與餐廳的數學座標 (Embedding Vectors)。但在此時，我們遇上了一個殘酷的現實：因為 Gemini API 額度的枯竭，**90% 的小店 (包含三民站附近的獨立咖啡館)** 都只被配給了代表無效的假向量 `[0.001] * 3072` (Dummy Vector)。

*   ⚙️ **系統日誌 (ChromaDB 搜索中)**:
    `➤ 帶著 "咖啡" 的數學特徵進入向量空間尋找 Top 500。`
    `➤ [警告] 三民站周邊的在地咖啡館失去語意特徵，無法匹敵大安區擁有真實向量的 "星巴克" 與名店！`
    `➤ ChromaDB 繳回的名單中，**三民站的在地咖啡館全軍覆沒**，清一色全是遙遠的東區連鎖店。`

如果我們只依賴向量庫 (就像傳統的 RAG 教程所教的一樣)，使用者將會看到系統要他從三民站走 3 公里去大安區喝咖啡的荒謬結果。

---

## 第三站：雙引擎出擊 —— SQLite 的在地神援 (Local Bounding Box)

就在三民站的在地咖啡館即將被系統拋棄之際，資料流被我們的 **Hybrid Retriever (混和檢索器)** 硬生生地攔截。

系統偵測到了真實的物理座標 (南京三民站)，立刻啟動了第二顆引擎 —— **SQLite Geographic Bounding Box**：

*   ⚙️ **系統日誌**:
    `➤ 啟動救援機制！以使用者 GPS 為圓心，畫出 500 公尺的物理方圓空間。`
    `➤ 對 SQLite 下達無情指令：SELECT 所有在這個方塊內的 food 店家！`
    `➤ 瞬間回傳 212 筆在地商家。`
    `➤ 【Poor Man's Semantics 啟動】掃描這 212 家店的招牌與描述。`
    `➤ 發現目標！"Daychill Specialty Coffee Co." 的 OSM 標籤中含有 "coffee_shop"！`
    `➤ 系統強制賦予它【0.25】的神級語意分數，讓它重新獲得參賽資格！`

SQLite 就像是在地的里長伯，它不管你在高維空間的數學特徵有多差，只要你真的開在三民站旁邊而且有賣咖啡，它就霸道地把你拉進候選名單中。

---

## 第四站：照妖鏡與守門員 —— 閾值計分與在地逆襲 (Threshold-Based Ranking)

現在，會議桌上同時坐著 ChromaDB 找來的大安區神級名店，以及 SQLite 救回來的在地小店。
最後的生死戰交給了我們的 **Threshold-Based Sorter (閾值排序器)**。

我們的評分公式是：`語意分數 + max(0, 距離 - 0.5)`。

*   🎯 **真實模擬結果**:
    1.  🥇 `[Daychill Specialty Coffee Co.]` (在地小店)
        - 語意: 0.25 (由 SQLite 關鍵字配給)
        - 距離: 0.15 km (小於 0.5km，距離懲罰 = 0)
        - **總分 = 0.2500** 🏆
    2.  🥈 `[遠方的大安區高級名店]` (Chroma菁英)
        - 語意: 0.15 (完美的數學向量匹配)
        - 距離: 2.15 km (超過 0.5km，距離懲罰 = 1.65)
        - **總分 = 1.8000** 慘遭淘汰！

**【技術檢討：為何我們需要雙料庫合擊】**
這就是這套架構最迷人的地方。**ChromaDB 保證了推薦的下限不會離題，而 SQLite 加上零懲罰安全區理論，則保證了推薦的上限絕對能「在地化」**。我們成功治癒了 RAG 系統常見的「資訊密度迷航症」。

---

## 終站：文字的魔術師 —— 導覽生成器 (Guide Generator)

最後，成功逆襲的在地小店 `Daychill Specialty Coffee Co.`、`1/2咖啡` 被送回給了 LLM 導遊。

導遊結合了使用者的情境，將這份歷經雙引擎淘汰賽的終極名單，化為溫暖的文字：

> 🪄 **System Output (模擬)**: 
> 「剛出捷運站想歇歇腳嗎？為您精選了三民站方圓 200 公尺內的在地獨立咖啡館。首推隱身巷弄的 `Daychill Specialty Coffee Co.`，不用走遠，轉個彎就能享受絕佳的特調咖啡與蛋糕，讓您立刻放鬆身心！」

伴隨著這段文字，前端渲染出了精美的導航卡片。一筆原本差點因為 API Quota 枯竭而永遠沉睡在資料庫深處的冰冷 OSM 資料，就這樣透過我們建立的雙軌搜救機制，成功來到了使用者的眼前。這套抗打擊的 Hybrid RAG 架構，正式宣告完工。
